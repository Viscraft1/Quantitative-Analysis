# 量化选股系统设计方案

## 1. 目标与边界

### 1.1 产品目标
- 打造**高频稳定的印钞机**，而非仅仅是“能跑通的爬虫软件”。
- 追求**极致的算力分配**与**模型降维打击**。
- 核心指标：**10倍执行效率** + **10倍盈利潜力**。

### 1.2 MVP 范围
- 市场范围：A 股。
- 数据来源：**核心依赖 AkShare** (Tick/盘口/实时数据)，**辅助 Baostock / Pytdx**，弱化 Tushare (因积分门槛与数据同质化)。
- 因子范围：**微观结构 (Tick/OrderFlow)** + 技术面 + 基本面。
- 策略方向：从日频选股转向 **盘中/尾盘 Tick 级信号** 捕捉。
- 调度：`Cron + Python CLI`。

### 1.3 非目标
- 暂不进行全市场 Tick 级历史回溯（数据量过大），只保留关键时间点快照。

## 3. 技术选型建议

### 3.1 推荐组合
- 后端：`Python 3.12`、`FastAPI`、`SQLAlchemy`、`Pydantic`。
- 任务调度：MVP 使用 `Cron + CLI`；进阶升级到 `Airflow`。
- 缓存/消息：`Redis`。
- 时序与关系数据：`PostgreSQL + TimescaleDB`。
- 向量计算：**`Polars`** (强制替代 Pandas，提升 I/O 和计算效率)。
- 机器学习：**`LightGBM` / `PyTorch`** (用于非线性 Alpha 挖掘)。
- 异步框架：**`asyncio` + `aiohttp`** (用于高频并发数据获取)。
- 非结构化文本：MVP 先存 `PostgreSQL JSONB`；文本量大后再拆 `MongoDB`。
- 前端：`React + Ant Design + ECharts / Lightweight Charts`。
- 指标库：`pandas-ta` 优先；如后续性能要求更高再引入 `TA-Lib`。
- 数据处理：`pandas + numpy + pyarrow`。
- 日志监控：`structlog` 或标准 logging，配合 `Prometheus + Grafana`。

### 3.2 关键升级点
- **废弃单线程 Pandas 循环**：在数据获取和处理层，强制使用 `multiprocessing` 和 `Polars`。
- **下沉核心计算**：对于极度耗时的因子计算，考虑使用 C++ 扩展 (pybind11)。
- **数据库批处理**：强制使用 `executemany` 或 `COPY`，禁止单条 `INSERT`。

## 4. 系统模块设计

## 4.1 数据获取层 Data Acquisition

### 4.1.1 目标
- 统一对接多种外部数据源。
- 给上层输出统一字段模型，屏蔽不同 API 的字段差异。
- 支持 API 主源 + 本地文件补源 + 爬虫辅助源。

### 4.1 数据获取层 Data Acquisition

#### A. 行情适配器 `market_data_adapters`
- `akshare_adapter.py`：**主战场**。
  - `stock_zh_a_spot_em`: 获取全市场实时快照 (盘口/五档/成交)。
  - `stock_zh_a_tick_tx_js`: 获取指定股票当日/历史 Tick 数据 (慎用，量大)。
  - `stock_zh_a_hist`: 日线/分钟线数据。
- `baostock_adapter.py`: 免费的历史数据下载备份，作为 AkShare 的交叉验证。
- `pytdx_local_adapter.py`: 读取本地通达信 `.day` / `.lc1` (1分钟) / `.lc5` (5分钟) 文件。
- `tushare_adapter.py`: (降级/备用) 仅用于获取基础信息或特定财务指标。

#### B. 数据存储模型
- `TickData / MarketSnapshot`: 存储关键时间点（如 09:25, 09:30, 14:50, 15:00）的盘口快照。
  - 字段：`symbol`, `time`, `price`, `bid1_price`, `bid1_vol`, `ask1_price`, `ask1_vol`, ...
- `DailyBar`: 传统的日线数据 (OHLCV)。
- `IntradayBar`: 分钟线数据 (1min/5min)。

#### B. 文本采集器 `content_collectors`
- `eastmoney_news_spider.py`
- `sina_finance_spider.py`
- `ths_report_spider.py`

#### C. 统一数据契约 `schemas`
- `Security`
- `DailyBar`
- `AdjustFactor`
- `FinancialIndicator`
- `NewsArticle`
- `SentimentScore`

### 4.1.3 获取层设计原则
- 所有源都输出统一 symbol 格式，例如：`600000.SH`、`000001.SZ`。
- 保留 `source`、`fetched_at`、`version` 字段，方便回溯。
- 原始响应落盘到 `raw` 区，便于排错与重放。
- 限流、重试、熔断统一放在 adapter 基类中。

### 4.1.4 现有脚本如何纳入
- 现有 `import tdxpy.py` 可改造成 `pytdx_local_adapter.py`。
- 现有 `search.py` 可抽取为最初版 `screening_demo.py`，保留作策略原型验证。

### 4.2.3 ETL 关键流程

#### 复权处理
- 原始价格与复权因子分开存储。
- 查询或计算指标时生成前复权序列，避免因为分红送股造成技术指标失真。
- 保留 `raw` 和 `qfq` 两套价格视图，方便审计。

#### 缺失值处理
- 停牌日不伪造 OHLC。
- 对连续型指标如市值、财务因子允许按最近一期公告前向填充。
- 对成交量、换手率等交易性字段不做盲目填充。

#### 异常值校验
- A 股普通股涨跌幅大于 10% 时标记异常，创业板/科创板需按板块规则判断。
- `high < low`、`close <= 0`、`volume < 0` 等直接拦截。
- 单源异常时与第二数据源交叉校验。

#### 去重与幂等
- 每次任务按 `trade_date + symbol + source` 做幂等写入。
- 任务失败后允许重跑，不生成重复快照。

## 4.3 核心计算与选股逻辑层

### 4.3.1 模块划分
- `factor_engine`：计算技术因子、基本面因子、资金因子、情绪因子。
- `filter_engine`：执行硬过滤条件。
- `ranking_engine`：标准化、加权打分、排序。
- `explanation_engine`：生成“高分原因”。

### 4.3.2 技术面因子示例
- `ma20_distance`：收盘价相对 20 日均线偏离度。
- `ma20_slope`：20 日均线斜率。
- `macd_hist_trend`
- `rsi_14`
- `boll_position`
- `vol_ratio_5`
- `momentum_20`

### 4.3.3 K线形态识别 (借鉴 InStock)
- **核心逻辑**：引入 `TA-Lib` 的 Pattern Recognition 功能，识别 61 种经典 K 线形态。
- **形态举例**：
  - **看涨**：早晨之星 (Morning Star)、红三兵 (Three White Soldiers)、锤头 (Hammer)。
  - **看跌**：三只乌鸦 (Three Black Crows)、黄昏之星 (Evening Star)、射击之星 (Shooting Star)。
- **信号处理**：
  - 将识别结果（100/0/-100）作为独立因子存入 `fact_kline_pattern` 表。
  - 在选股时，可作为过滤条件（如“必须出现看涨形态”）或加分项（如“出现红三兵 +5 分”）。

### 4.3.4 基本面因子示例
- `pe_ttm_percentile_industry`
- `pb_percentile_industry`
- `roe_ttm`
- `revenue_yoy`
- `net_profit_yoy`
- `debt_ratio`

### 4.3.5 资金面因子示例
- `turnover_rate`
- `amount_ratio_5`
- `main_inflow`（后续可接）
- `northbound_change`（后续可接）

### 4.3.6 情绪面因子示例
- 新闻标题情感分。
- 股吧帖子情绪均值。
- 利空/利好关键词计数。

### 4.3.7 硬过滤条件
- 剔除 `ST/*ST`。
- 剔除上市不足 `120` 天的新股。
- 剔除日均成交额过低个股。
- 剔除换手率低于 `1%` 的僵尸股。
- 剔除总市值低于 `20` 亿元个股。
- 剔除停牌、退市整理、数据不完整个股。

### 4.3.8 评分模型建议

**升级方案：从线性加权转向机器学习模型**

- **Phase 1 (MVP)**: 维持线性加权模型，作为 Baseline。
  ```text
  TotalScore = 0.4 * Tech + 0.35 * Fund + 0.15 * Flow + 0.1 * Sentiment
  ```
- **Phase 2 (ML)**: 引入 `LightGBM` 或 `XGBoost`。
  - **Label**: 未来 5 日/10 日的超额收益率 (vs 中证500)。
  - **Features**: 各类因子原始值、行业哑变量、市值对数。
  - **训练方式**: 滚动窗口训练 (Rolling Window)，例如用过去 3 年数据训练，预测下个月。

#### 标准化方法
- 因子先做 winsorize 截尾。
- 再做行业内 Z-Score 标准化，降低行业偏差。
- 对越小越好的因子做反向处理，例如 `PE/PB`。

### 4.3.9 微观结构因子 (Alpha 挖掘)
- **大单净流入占比**: 开盘后 30 分钟内，超大单与大单的主动买入净额占成交额比例。
- **高频波动率偏度**: 日内 1 分钟线收益率分布的偏度，衡量尾部风险。
- **委买委卖失衡度**: 盘口挂单的买卖压力差。

### 4.3.10 回测接口预留
- **核心原则**: 回测前置。在 Phase 1 即建立回测能力，而非等到最后。
- **关键参数**:
  - `slippage`: 滑点 (如 0.1%)。
  - `commission`: 佣金 (如 0.03%)。
  - `limit_move`: 涨跌停限制处理（无法买入/卖出）。
- **输出**: 年化收益、最大回撤、夏普比率、胜率、换手率。
- **第一阶段**: 实现日频向量化回测框架。

## 4.4 调度与自动化层

### 4.4.1 MVP 调度方案
- 使用 `Cron` 或 Windows 任务计划程序。
- 每个任务调用一个独立 CLI：
  - `fetch_daily_bars`
  - `fetch_financials`
  - `clean_daily_data`
  - `compute_factors`
  - `run_screening`
  - `publish_results`

### 4.4.2 进阶调度方案
- 升级到 `Airflow` 后，DAG 示例：

```text
sync_trade_calendar
  -> fetch_security_master
  -> fetch_daily_bars
  -> fetch_adjust_factors
  -> fetch_financials
  -> fetch_news
  -> clean_and_validate
  -> compute_factors
  -> run_filters_and_rank
  -> publish_result
  -> notify
```

### 4.4.3 推荐触发时间 (执行前置)
- **盘中预警 (14:50)**: 
  - 拉取实时快照行情。
  - 结合昨收盘因子，预估今日收盘信号。
  - 目的：尾盘抢筹，提高资金利用率。
- **早盘竞价 (09:25)**:
  - 监控集合竞价异动。
- **日终归档 (16:30)**:
  - 完整跑批，生成次日备选池。
  - 更新历史数据库。

### 4.4.4 失败处理
- 任务自动重试 `3` 次。
- 核心表写入前做事务控制。
- 失败记录落 `job_run_log`。
- 告警方式：邮件、企业微信、钉钉任选其一。

## 6. 关键数据流设计

### 6.1 日终主流程
1. 拉取交易日历，确认今天是否为交易日。
2. 拉取股票基础资料与状态。
3. 拉取日线行情、复权因子、换手率、成交额。
4. 拉取财务指标。
5. 拉取新闻标题和研报摘要。
6. 标准化代码、去重、校验。
7. 生成前复权价格序列。
8. 计算因子快照。
9. 执行硬过滤。
10. 计算综合得分并排名。
11. 将结果写入 `screening_result`。
12. 刷新 API 缓存并通知前端。

### 6.2 查询主流程
1. 前端请求日榜或个股详情。
2. API 优先从缓存读取。
3. 缓存 miss 时查 `serving / mart` 表。
4. 返回结构化 JSON。

## 8. 安全、审计与可维护性

### 8.1 安全
- API 仅暴露必要查询接口。
- 敏感配置放环境变量或 `.env`。
- 外部源 token 做权限隔离。

### 8.2 审计
- 每次筛选结果保留 `strategy_version`。
- 关键因子计算结果留快照。
- 所有原始数据保留 `source` 和 `fetched_at`。

### 8.3 可维护性
- adapter、factor、filter 均采用插件化接口。
- 所有任务必须幂等。
- 指标计算与 API 查询解耦。

## 9. 性能与数据质量建议

### 9.1 性能优化 (工程架构升级)
- **并发处理**: 在 `Data Acquisition` 层全面废弃单线程循环。使用 `asyncio` 或 `multiprocessing` 并发拉取数据。
- **向量化计算**: 引入 `Polars` 或优化 `Pandas` 逻辑，严禁在 Python 中使用 `for row in dataframe` 循环。
- **数据库批处理**: 写入操作强制使用 `executemany` 或 `COPY`，拒绝单条 `INSERT`。

### 9.2 数据质量 (幸存者偏差)
- **退市股票**: 历史数据必须包含已退市股票。
- **ST 股票**: 保留 ST 期间的数据，但在选股时根据策略标记或过滤，而不是直接物理删除。

### 9.3 数据规模预估
- A 股约 `5000+` 支股票。
- 5 年日线约 `5000 × 1200` 级别，单机 PostgreSQL 足够承载。
